# Backend — Architecture

Companion to `PLAN.md`. This doc covers the technical layout: module structure, package layout,
database schema, request flows, and security model.

---

## 1. Module Structure

```
backend/
├── build.gradle.kts                    # Spring Boot + Kotlin + JPA dependencies
├── settings inherited from root
├── docker-compose.yml                  # Local: postgres+postgis on :5432
├── docker-compose.prod.yml             # Prod: caddy → spring → postgres + photo volume
├── Dockerfile                          # Multi-stage Gradle JIB-style build
├── .env.example                        # Documents required env vars (no secrets)
├── scripts/
│   └── translate-seed.main.kts         # One-time: xlsx → Anthropic API → seed.json
└── src/
    ├── main/
    │   ├── kotlin/com/menudeldia/
    │   │   ├── MenuDelDiaApplication.kt
    │   │   ├── config/                 # Spring beans, security, CORS, JPA
    │   │   ├── auth/                   # Sign-in flow (controller, JWT, filter)
    │   │   ├── restaurant/             # Domain: entity, repo, service, controller, dto, mapper
    │   │   ├── places/                 # Google Places client + enrichment service
    │   │   ├── photo/                  # Photo download, storage, streaming controller
    │   │   ├── seed/                   # Seeder (reads seed.json on first boot)
    │   │   ├── geo/                    # Distance, bounding-box helpers (if needed beyond PostGIS)
    │   │   └── common/                 # Errors, validation, ratelimit, web config
    │   └── resources/
    │       ├── application.yml
    │       ├── application-dev.yml
    │       ├── application-prod.yml
    │       ├── db/migration/V1__init.sql            # Flyway
    │       ├── db/migration/V2__add_places_fetched_at.sql
    │       └── seed.json                            # Generated by translate script
    └── test/
        └── kotlin/com/menudeldia/
            ├── restaurant/                          # Repo + controller tests (Testcontainers)
            ├── places/                              # Mock Google Places HTTP
            ├── auth/                                # JWT + ID token verification
            └── support/                             # Test fixtures, base classes
```

**Why package-by-feature, not by-layer:** keeps related code together (the `restaurant` package owns
its entity, repo, service, controller, and DTOs). Cross-cutting code lives in `common/`, `config/`,
`auth/`. New contributors can find everything for a feature in one folder.

## 2. Layered Responsibilities (within a feature package)

```
HTTP request
   ↓
Controller (HTTP / DTO ↔ domain)
   ↓
Service (business logic, orchestrates repos + external calls)
   ↓
Repository (Spring Data JPA — DB access only)
   ↓
Database
```

Rules (mirrors root `CLAUDE.md` "Code Quality" section):

- **Controllers** validate input, map DTO ↔ domain, return `ResponseEntity`. No business logic.
- **Services** own the logic. They never see DTOs and never touch HTTP.
- **Repositories** are JPA-only. No business logic. No conditional includes.
- **Entities** model the DB. They never leak through controllers — always mapped to a DTO.
- **DTOs** are versioned per API version (`v1`). Renaming a field on an entity does not break the
  wire format.

## 3. Database Schema

PostgreSQL 16 + PostGIS 3.4. Migrations via Flyway (`db/migration/V*__*.sql`).

### `restaurants`

| Column              | Type                        | Notes                                                               |
|---------------------|-----------------------------|---------------------------------------------------------------------|
| `id`                | `UUID PRIMARY KEY`          | Generated server-side                                               |
| `name`              | `TEXT NOT NULL`             | From xlsx                                                           |
| `address`           | `TEXT`                      | From Google Places (enriched)                                       |
| `lat`               | `DOUBLE PRECISION NOT NULL` | From Google Places (or geocoded from Place ID)                      |
| `lng`               | `DOUBLE PRECISION NOT NULL` | Same                                                                |
| `geom`              | `GEOGRAPHY(Point, 4326)`    | Generated from lat/lng; GiST-indexed                                |
| `phone`             | `TEXT`                      | From xlsx primary, Places fallback                                  |
| `website`           | `TEXT`                      | From xlsx primary, Places fallback                                  |
| `google_place_id`   | `TEXT UNIQUE`               | From xlsx                                                           |
| `cuisine_type`      | `TEXT`                      | Mapped enum (e.g., `ASIAN`, `MEDITERRANEAN`, `SPANISH`, ...)        |
| `cuisine_emoji`     | `TEXT`                      | Per-cuisine emoji shown on map markers                              |
| `description_es`    | `TEXT`                      | Translated from Hebrew at seed time (or from Google Places)         |
| `description_en`    | `TEXT`                      | Translated from Hebrew at seed time                                 |
| `menu_price`        | `NUMERIC(6,2)`              | EUR; from xlsx `normal price`                                       |
| `currency`          | `CHAR(3) DEFAULT 'EUR'`     |                                                                     |
| `price_includes_es` | `JSONB`                     | `["Primero", "Segundo", "Bebida"]` — derived from xlsx Menu Details |
| `price_includes_en` | `JSONB`                     | `["First", "Main", "Drink"]`                                        |
| `weekday_hours`     | `JSONB`                     | `{"mon":"12:30-16:00","tue":"...","fri":"..."}` — Mon–Fri only      |
| `opening_hours`     | `JSONB`                     | Full week from Google Places (for detail screen)                    |
| `photo_count`       | `INT DEFAULT 0`             | Number of photos cached locally                                     |
| `places_fetched_at` | `TIMESTAMPTZ`               | Last successful Google Places enrichment; NULL = never fetched      |
| `created_at`        | `TIMESTAMPTZ DEFAULT now()` |                                                                     |
| `updated_at`        | `TIMESTAMPTZ DEFAULT now()` | Updated via trigger or `@PreUpdate`                                 |

**Indexes:**

- `idx_restaurants_geom` GIST on `geom`
- `idx_restaurants_places_fetched_at` BTREE on `places_fetched_at` (for "find stale rows")
- `uq_restaurants_google_place_id` UNIQUE on `google_place_id`

**Why no `menus` table:** the menú del día is the same Mon–Fri for these restaurants in v1 (per
locked decision in PLAN.md). We promote daily-menu modeling to v2.

### `users`

| Column         | Type                        | Notes                                           |
|----------------|-----------------------------|-------------------------------------------------|
| `id`           | `UUID PRIMARY KEY`          |                                                 |
| `provider`     | `TEXT NOT NULL`             | `google` or `apple`                             |
| `external_id`  | `TEXT NOT NULL`             | The `sub` claim from the ID token               |
| `email`        | `TEXT`                      | May be null (Apple "private relay" hides email) |
| `display_name` | `TEXT`                      |                                                 |
| `avatar_url`   | `TEXT`                      |                                                 |
| `created_at`   | `TIMESTAMPTZ DEFAULT now()` |                                                 |
| `last_login`   | `TIMESTAMPTZ`               |                                                 |

**Indexes:**

- `uq_users_provider_external_id` UNIQUE on `(provider, external_id)`

## 4. API Surface (v1)

All routes prefixed with `/api/v1/`. Public unless marked **[auth]**.

| Method | Path                           | Purpose                                          |
|--------|--------------------------------|--------------------------------------------------|
| `GET`  | `/health`                      | Liveness probe (returns `OK`)                    |
| `GET`  | `/restaurants`                 | List nearby restaurants (query params below)     |
| `GET`  | `/restaurants/{id}`            | Restaurant detail                                |
| `GET`  | `/restaurants/{id}/photos/{n}` | Stream cached photo bytes (n = 0..photo_count-1) |
| `POST` | `/auth/google`                 | Exchange Google ID token → session JWT           |
| `POST` | `/auth/apple`                  | Exchange Apple identity token → session JWT      |
| `GET`  | `/me`                          | **[auth]** Current user profile                  |

**`/restaurants` query params:**

- `lat` (required, double)
- `lng` (required, double)
- `radius` (meters, default 2000, max 10000)
- `q` (search by name, dish, cuisine — ILIKE on name/cuisine_type)
- `openNow` (bool — checks current time vs `weekday_hours[today]`)
- `cuisine` (cuisine enum value, repeatable)
- `minPrice`, `maxPrice` (€)

**Response shape** matches the existing client `Restaurant` domain model — see
`shared/.../domain/model/Restaurant.kt` and `data/remote/dto/RestaurantDto.kt`. We adapt to that
contract; we don't redefine it.

## 5. Request Flow — `GET /restaurants?lat&lng&radius`

```
Client
  ↓
RateLimitFilter (Bucket4j)         — 60/min per IP, else 429
  ↓
SecurityFilterChain (Spring)        — public, no auth required
  ↓
RestaurantController.list()
  ↓ DTO validation (@Valid on query params)
  ↓
RestaurantService.findNearby(lat, lng, radius, filters)
  │   ├── PlacesEnrichmentService.refreshStale(maxRows = 5)   ← async-ish: triggers if any rows in radius are >1h old
  │   └── RestaurantRepository.findNearby(...)                ← native PostGIS query: ST_DWithin(geom, point, radius)
  ↓
List<Restaurant> (entities)
  ↓ RestaurantMapper.toDto() — adds distanceMeters, isOpenNow
  ↓
List<RestaurantDto>
  ↓
Controller wraps in { restaurants: [...] } and returns 200
```

**Key invariants:**

- The DB query does the geo filter — no in-memory filtering.
- `isOpenNow` is computed in the mapper from `weekday_hours` + server's `Europe/Madrid` clock (not
  client's).
- Photo URLs in the response point to **our** `/api/v1/restaurants/{id}/photos/{n}` — never to
  Google.

## 6. Google Places Enrichment Flow

Triggered lazily on `GET /restaurants` when one or more rows in the result set have
`places_fetched_at < now() - INTERVAL '1 hour'` (or NULL).

```
RestaurantService.findNearby()
  ↓
  ↓ (rows fetched from DB)
  ↓
PlacesEnrichmentService.refreshIfStale(rows)
  │   For up to 5 stalest rows in parallel (with semaphore):
  │     ├── GooglePlacesClient.placeDetails(google_place_id)
  │     │     → GET https://places.googleapis.com/v1/places/{placeId}
  │     │       fields: location, photos, regularOpeningHours, formattedAddress, internationalPhoneNumber, websiteUri
  │     │       headers: X-Goog-Api-Key, X-Goog-FieldMask
  │     ├── PhotoStorageService.downloadPhotos(placeId, photoNames)
  │     │     → For each photo not already on disk:
  │     │         GET https://places.googleapis.com/v1/{photoName}/media?maxHeightPx=800
  │     │         Save bytes to /data/photos/{restaurantId}/{n}.jpg
  │     └── RestaurantRepository.save(updatedRow)
  │           sets address, lat, lng, opening_hours, photo_count, places_fetched_at = now()
  ↓
Updated rows returned to caller (next request gets fresh data)
```

**Caching invariants:**

- 1-hour TTL keyed on `places_fetched_at`.
- "Stale storm" protection: per request, refresh max 5 rows; rest stay served from DB until they're
  picked next time. A flat 60s in-memory dedupe map prevents two concurrent requests from refreshing
  the same row.
- A daily quota check (counter in DB or Caffeine) fails open if Google returns 429 — we serve the
  stale row.

## 7. Authentication Flow

### Sign-in (`POST /auth/google` shown; `/auth/apple` is identical structure)

```
Client → POST /api/v1/auth/google { idToken: "<google ID token>" }
  ↓
AuthController.signInWithGoogle()
  ↓
GoogleIdTokenVerifier (google-api-client lib)
  │ - validates against Google's JWKS (cached, auto-rotated)
  │ - checks `aud` matches our OAuth client ID
  │ - returns claims: sub, email, name, picture
  ↓
UserService.upsert(provider="google", externalId=sub, email, name, avatarUrl)
  ↓
JwtService.issue(userId)
  │ HS256, 30-day expiry, claims: { sub: userId, iat, exp }
  ↓
Response: 200 { accessToken: "<our JWT>", user: { id, email, name, avatar } }
```

Apple sign-in differs only in the verifier (Apple's JWKS at `https://appleid.apple.com/auth/keys`) —
same upsert, same JWT issuance.

### Authenticated requests (`/api/v1/me`)

```
Client → GET /api/v1/me
         Authorization: Bearer <our session JWT>
  ↓
JwtAuthFilter
  │ - extracts token, validates signature + expiry
  │ - on success: SecurityContext.setAuthentication(userPrincipal)
  │ - on failure: 401 (do not call downstream)
  ↓
MeController.me() — reads SecurityContext, returns user DTO
```

**Why our own JWT instead of forwarding Google/Apple tokens:**

- Apple ID tokens are short-lived (~10 min); requiring re-validation on every request would burden
  the client with refresh logic.
- Our JWT is signed with our key; it's the only thing that grants access to our APIs.

## 8. Security Model

| Concern            | Control                                                                                                                                           |
|--------------------|---------------------------------------------------------------------------------------------------------------------------------------------------|
| Transport          | HTTPS-only; Caddy reverse proxy auto-issues Let's Encrypt cert; HSTS via Spring config                                                            |
| Headers            | `Strict-Transport-Security`, `X-Content-Type-Options`, `X-Frame-Options`, CSP                                                                     |
| CORS               | Allow `https://app.menu-del-dia.example` + dev hosts; disallow `*` in prod                                                                        |
| Authentication     | Spring Security + custom `JwtAuthFilter` for our session JWT (HS256)                                                                              |
| Authorization      | `/api/v1/me/**` requires authenticated principal; everything else public                                                                          |
| Input validation   | Bean Validation (`@Valid`, `@NotNull`, `@Size`, `@Min`, `@Max`) on all DTOs                                                                       |
| SQL injection      | JPA prepared statements + named parameters in any native PostGIS query                                                                            |
| Rate limiting      | Bucket4j filter — 60 req/min per IP on read endpoints, 10 req/min on `/auth/*`                                                                    |
| Secrets            | Env vars only: `DB_PASSWORD`, `GOOGLE_PLACES_API_KEY`, `JWT_SIGNING_KEY`, `GOOGLE_OAUTH_CLIENT_ID`, `APPLE_TEAM_ID`. Documented in `.env.example` |
| Logging            | `logback-json` for structured logs; never log tokens, secrets, full request bodies                                                                |
| Dependency hygiene | `gradle dependencyUpdates` weekly; `spring-boot-starter-actuator` patched promptly                                                                |

**Out of scope for v1** (deferred): API access logs, admin role, IP geo-blocking, audit trail of
writes.

## 9. Configuration

`application.yml` (env-driven):

```yaml
server:
  port: 8080

spring:
  datasource:
    url: ${DB_URL:jdbc:postgresql://localhost:5432/menudeldia}
    username: ${DB_USER:menudeldia}
    password: ${DB_PASSWORD}
  jpa:
    hibernate:
      ddl-auto: validate     # Flyway owns schema, JPA validates only
    properties.hibernate.dialect: org.hibernate.dialect.PostgreSQLDialect
  flyway:
    locations: classpath:db/migration

menudeldia:
  google:
    places-api-key: ${GOOGLE_PLACES_API_KEY}
    places-cache-ttl: PT1H
    places-refresh-batch-size: 5
  auth:
    google-client-id: ${GOOGLE_OAUTH_CLIENT_ID}
    apple-client-id: ${APPLE_BUNDLE_ID}
    jwt-signing-key: ${JWT_SIGNING_KEY}
    jwt-ttl: P30D
  photos:
    storage-root: ${PHOTOS_DIR:/data/photos}
  rate-limit:
    read-rpm: 60
    auth-rpm: 10
```

Profiles: `dev` (local Docker Compose), `test` (Testcontainers), `prod` (real env vars, no defaults
for secrets).

## 10. Local Dev Loop

```bash
# 1. Spin up Postgres + PostGIS
docker compose up -d db

# 2. Run translation once (writes seed.json)
kotlin scripts/translate-seed.main.kts \
  --xlsx ../resturant-list.xlsx \
  --out src/main/resources/seed.json \
  --anthropic-key $ANTHROPIC_API_KEY

# 3. Boot the app (auto-runs Flyway, then seeds if DB empty)
./gradlew :backend:bootRun

# 4. Smoke test
curl 'http://localhost:8080/api/v1/restaurants?lat=41.3851&lng=2.1734&radius=2000' | jq
```

Tests use Testcontainers to stand up Postgres+PostGIS per test class — no shared mutable state
between tests.

## 11. Open Items for Phase 1 Spec

These are the small calls to make as we start coding (not blockers):

- Cuisine enum mapping table (Hebrew → enum) — define in seed script with manual review.
- Photo size — `maxHeightPx=800` fits both list (thumbnail) and detail screen; revisit if photos
  look bad on tall phones.
- Should `/restaurants/{id}/photos/{n}` set long `Cache-Control: max-age` headers? Yes — photos are
  immutable per `places_fetched_at` cycle. Use ETag = `places_fetched_at + n`.
- Distance calculation: PostGIS `ST_Distance(geom, point)` returns meters when both are GEOGRAPHY —
  use it directly in the SELECT.
